# AI小说平台拆书系统 - 产品原型与UI设计

## 1. 文档目标

本文档用于补齐拆书系统在步骤 3 的正式产物，独立承载以下内容：

1. 产品原型说明
2. 页面结构与信息架构
3. 关键交互规则
4. 视觉与组件规范
5. 可复用前端骨架约束

说明：
- 本文档不替代 PRD、功能架构图、产品流程图。
- 本文档仅覆盖拆书系统，不包含写作系统。
- 本文档以当前已实现的 Web 前端骨架为基础进行规范化沉淀。

## 2. 设计目标

### 2.1 用户目标
用户能够完成以下任务：

1. 上传小说文件并填写基础信息
2. 发起拆书处理并实时查看处理阶段
3. 在处理失败时快速定位错误阶段和错误信息
4. 在书库中查看历史拆书结果
5. 在详情页中查看世界纲 / 大纲 / 粗纲 / 章纲
6. 对生成结果进行查看、复制、导出

### 2.2 设计原则

1. **过程透明**：每个处理阶段必须可见
2. **错误可定位**：失败时必须显示阶段与错误内容
3. **结构清晰**：纲层级遵循“世界纲 → 大纲 → 粗纲 → 章纲”
4. **移动优先兼容桌面**：移动端单列，桌面端双栏/多栏
5. **骨架可复用**：上传、进度、纲树、Toast、设置模块可独立复用
6. **与后端契约一致**：UI阶段命名、字段命名尽量与 API / 状态模型统一

## 4. 页面原型说明

## 4.1 上传页

### 页面目标
完成文件导入、书籍信息补充、处理启动与实时监控。

### 区块结构
1. 顶部页面标题与说明
2. 文件上传区
3. 书籍信息表单
4. 操作按钮区
5. 处理进度区
6. 错误信息区

### 原型结构
```text
┌──────────────────────────────┐
│ 上传小说文件                  │
│ 支持 epub/txt/doc/docx/pdf   │
├──────────────────────────────┤
│ [拖拽上传区域 / 点击选择]     │
│ 格式徽标                      │
├──────────────────────────────┤
│ 书名 [__________]            │
│ 作者 [__________]            │
│ 简介 [__________]            │
├──────────────────────────────┤
│ [开始处理]   [重置]          │
├──────────────────────────────┤
│ 处理进度                      │
│ - 文件上传                    │
│ - 文本预处理                  │
│ - 章纲提取                    │
│ - 粗纲生成                    │
│ - 大纲生成                    │
│ - 世界纲生成                  │
├──────────────────────────────┤
│ 错误信息列表                  │
└──────────────────────────────┘
```

### 交互规则
- 选择文件后，自动根据文件名填充书名。
- 文件通过校验后才展示表单。
- 点击“开始处理”后：
  - 上传区保留当前文件状态
  - 进度区展开
  - 开始按钮禁用
- 任一阶段失败时：
  - 对应阶段标红
  - 错误信息进入错误列表
- 处理完成后：
  - 给出成功提示
  - 跳转详情页或允许用户手动查看详情

---

## 4.2 书库页

### 页面目标
展示拆书历史记录，并支持查看、筛选、删除。

### 区块结构
1. 搜索框
2. 状态筛选器
3. 书籍卡片网格

### 原型结构
```text
┌──────────────────────────────┐
│ 搜索 [____________] [状态▼]  │
├──────────────────────────────┤
│ [书籍卡片] [书籍卡片]         │
│ 标题/作者/状态/时间           │
│ [查看] [删除]                │
└──────────────────────────────┘
```

### 交互规则
- 默认按最近处理时间倒序。
- 查看进入详情页。
- 删除需二次确认。
- 空状态展示引导文案。

---

## 4.3 详情页

### 页面目标
用结构化方式展示纲结果，支持树形浏览、列表浏览、复制导出。

### 区块结构
1. 书籍元信息
2. 视图切换 Tab
3. 纲树视图
4. 列表视图
5. 导出与复制操作

### 原型结构
```text
┌────────────────────────────────────┐
│ 书名 / 作者 / 简介                  │
├────────────────────────────────────┤
│ [树图视图] [列表视图] [导出]        │
├────────────────────────────────────┤
│ 世界纲                              │
│  └─ 大纲                            │
│      └─ 粗纲                        │
│          └─ 章纲                    │
├────────────────────────────────────┤
│ 点击节点可查看全文 / 复制内容        │
└────────────────────────────────────┘
```

### 展示规则
- 层级顺序固定为：
  1. 世界纲 `worldOutline`
  2. 大纲 `mainOutline`
  3. 粗纲 `coarseOutline`
  4. 章纲 `chapterOutline`
- 树图用于表达上下级关系。
- 列表用于阅读详细内容。
- 后续图谱模式可在当前树图区域扩展，不改变主信息架构。

### 交互规则
- 点击节点可查看摘要与全文。
- 节点支持一键复制。
- 导出操作至少支持复制全部文本，后续再扩展文件导出。

## 4.4 设置页

### 页面目标
提供 API、模型、拆分参数与调试项配置。

### 区块结构
1. API Base URL
2. API Key
3. 模型名
4. temperature / max tokens
5. 章节切分大小
6. 保持段落完整
7. Mock 模式 / Debug 模式
8. 保存 / 重置

### 关键规则
- API Key 默认隐藏显示。
- Mock 模式默认开启，便于前端独立开发。
- 生产态必须走后端代理，不在前端持久暴露真实密钥。

## 5. 状态与阶段设计

## 5.1 标准阶段命名

前端统一使用以下处理阶段：

1. `fileUpload` 文件上传
2. `textPreprocess` 文本预处理
3. `chapterOutline` 章纲提取
4. `coarseOutline` 粗纲生成
5. `mainOutline` 大纲生成
6. `worldOutline` 世界纲生成

说明：
- 不再在 UI 层使用 `upload / decode / split / roughOutline` 作为主阶段命名。
- 如需表达“解码、清洗、去标题、切分”等内部处理步骤，统一归入 `textPreprocess`。

## 5.2 处理状态结构

推荐前端状态结构如下：

```json
{
  "isProcessing": false,
  "currentStage": null,
  "progress": 0,
  "fileUpload": { "status": "pending", "progress": 0 },
  "textPreprocess": { "status": "pending", "progress": 0 },
  "chapterOutline": { "status": "pending", "progress": 0, "total": 0, "completed": 0 },
  "coarseOutline": { "status": "pending", "progress": 0, "total": 0, "completed": 0 },
  "mainOutline": { "status": "pending", "progress": 0, "total": 0, "completed": 0 },
  "worldOutline": { "status": "pending", "progress": 0 }
}
```

## 6. 组件设计

## 6.1 FileUploader
职责：
- 选择文件
- 校验格式
- 校验大小
- 展示上传进度
- 抛出上传与错误事件

输入参数：
- `acceptedFormats`
- `maxFileSize`
- `onUpload`
- `onProgress`
- `onError`

约束：
- `acceptedFormats` 统一使用带点后缀，如 `.epub`、`.txt`

## 6.2 ProgressMonitor
职责：
- 展示标准阶段进度
- 展示阶段状态：等待中 / 进行中 / 完成 / 失败
- 展示错误列表

约束：
- 直接消费标准 `processingStatus`
- 不再依赖单独的 `stages` 嵌套结构

## 6.3 ToastManager
职责：
- 成功、失败、警告、提示消息反馈

## 6.4 纲展示组件
当前由详情页方法实现，后续可抽象为：
- `OutlineTree`
- `OutlineList`
- `OutlineDetailModal`

## 7. 视觉规范

## 7.1 颜色语义
- 主色：品牌蓝/紫系
- 成功：绿色
- 警告：橙色
- 错误：红色
- 中性：灰阶

## 7.2 状态语义
- `pending`：灰色
- `active`：主色高亮 + 动画
- `completed`：成功色
- `error`：错误色

## 7.3 响应式规则
- 移动端：单列布局，操作区纵向堆叠
- 平板：双栏为主
- 桌面端：上传/说明、树图/列表可横向布局

---

## 8. 可复用前端骨架约束

1. 页面入口集中在 `index.html`
2. 状态集中在 `js/state.js`
3. API 调用集中在 `js/api.js`
4. UI组件集中在 `js/components.js`
5. 页面编排与交互集中在 `js/app.js`
6. 样式集中在 `css/styles.css`

约束：
- 页面层不直接硬编码业务阶段别名
- 命名以 PRD / API / 架构文档为准
- Mock 数据结构优先贴近 OpenAPI 契约
- 新增组件时优先抽象，不在页面中堆叠大段重复模板
