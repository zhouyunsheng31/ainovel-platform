## 一、规划目标

本规划用于回答三个问题：

1. 当前项目下一步应该先做什么
2. 某一步完成后应该紧接着做什么
3. 哪些步骤可以并行推进，哪些步骤必须串行

本规划以 PRD 中定义的 MVP 功能为主线，不提前插入账号体系、导出功能、历史记录、写作系统等后续迭代项。

---

## 二、规划依据

### 2.1 PRD 的 MVP 范围

MVP 必须闭环的 8 个核心能力如下：

1. 文件上传
2. 文本预处理
3. 章纲提取
4. 粗纲生成
5. 大纲生成
6. 世界纲生成
7. 进度监控
8. 纲展示页面

### 2.2 当前审计结论摘要

当前项目状态不是“可直接发布”，主要结论如下：

| 类别 | 现状 |
|------|------|
| 文档 | 核心文档齐全，但完成状态表述不一致 |
| 后端 | API 主体可运行，21 个 pytest 用例通过 |
| 前端 | 存在启动即报错问题，不能视为可交付 |
| 门禁 | Ruff 未通过，OpenAPI 有 warning |
| 工作流测试 | LangGraph 工作流未被当前 pytest 套件真实覆盖 |
| Android | Gradle Wrapper 配置异常，无法在当前环境完成验证 |
| 安全 | 仓库中存在真实密钥暴露风险，必须优先处理 |

### 2.3 排序原则

所有工作按以下优先级排序：

1. 安全问题优先于功能问题
2. 启动阻断优先于体验优化
3. 可测试优先于可扩展
4. 联调完成优先于发布准备
5. MVP 闭环优先于新增需求

## 四、串行主链

以下是必须按顺序推进的主链路：

```text
P0 安全与阻断修复
  ↓
P1 质量门禁清零
  ↓
P2 工作流与测试补齐
  ↓
P3 前后端联调
  ↓
P4 MVP 回归验收
  ↓
P5 稳定性与构建收尾
  ↓
P6 发布准备
```

原因如下：

- 前端未修复启动错误前，无法做真实联调
- 工作流未进入正式测试前，不能声称后端核心链路已验证
- 门禁未通过前，不适合进入 PRD 验收或发布准备
- Android 构建未修复前，移动端交付不能闭环

## 五、并行泳道

在不破坏主链的前提下，以下任务可以并行推进：

| 并行组 | 可并行内容 | 前置条件 | 说明 |
|--------|------------|----------|------|
| A组 | 密钥清理 + lint 修复 | 无 | 两者互不依赖，可同时推进 |
| B组 | 工作流解耦 + OpenAPI 文档补全 | P1 完成 | 一个偏代码结构，一个偏契约文档 |
| C组 | API 联调 + WebSocket 联调 | P2 完成 | 同属真实联调，但可由不同人分工 |
| D组 | PRD 功能回归 + UI/响应式回归 | P3 基本稳定 | QA 可拆成功能和界面两条线 |
| E组 | Python 3.14 兼容治理 + 日志/观测增强 | P4 完成 | 同属工程质量增强，可并行 |
| F组 | Android Wrapper 修复 + 文档统一 | P4 完成 | 一条偏构建，一条偏交付材料 |

以下任务不建议并行：

- 前端启动修复 与 前端真实联调
- 工作流循环依赖未解 与 工作流测试编写
- 主链路未跑通 与 发布准备
- Android 构建异常 与 Android 交付验收

## 十二、结论

本项目的正确推进方式不是继续新增需求，而是：

1. 先消除安全和启动阻断
2. 再补齐核心链路测试
